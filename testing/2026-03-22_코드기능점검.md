## 3. 발견된 버그

### 🔴 [CRITICAL] attractionId 타입 불일치 — accessScore 미반영

**파일**: `backend/src/app.ts:79`

**현상**:
프론트엔드(`ScorePanel.tsx`)는 `/api/score/${attraction.id}`로 호출하는데,
`attraction.id`는 **UUID** 형식이다. 그러나 백엔드 score 엔드포인트는 `contentId`로 DB를 조회한다.

```typescript
// app.ts:79 — contentId로 조회
const attraction = await prisma.attraction.findFirst({
  where: { contentId: attractionId },  // ← attractionId가 UUID면 매칭 안 됨
  select: { accessScore: true }
});
```

**영향**:
- 프론트에서 UUID로 호출 → DB 조회 실패 → `accessScore = 0.5` fallback 고정
- contentId로 호출했을 때와 최종 점수 **최대 5점 차이** 발생
- 국립일제강제동원역사관 기준: UUID 호출 78.3점 vs contentId 호출 83.3점

**원인**: `id` (UUID)와 `contentId` (UC_SEQ 숫자 문자열) 두 식별자가 혼재

**수정 방법**: score 엔드포인트에서 `id`와 `contentId` 둘 다 조회하도록 수정 필요
```typescript
where: { OR: [{ contentId: attractionId }, { id: attractionId }] }
```

### 🟡 [MEDIUM] 지하철 전용 경로 시 waitTimeMin 고정 5분

**파일**: `backend/src/services/routeService.ts:54-58`

**현상**:
`extractFirstBusStopName()`은 `trafficType === 2` (버스)만 추출한다.
지하철만 경유하는 경로(`trafficType === 1`)는 항상 `busStopName = null` → `waitTimeMin = 5` 고정.

```typescript
const busSub = subPaths.find((sp: any) => sp.trafficType === 2); // 버스만
```

**영향**: 지하철 경로는 BusanBIMS 실시간 대기시간 미활용, 고정 5분 패널티 부여

### 🟡 [MEDIUM] BusanBIMS 정류소명 불완전 매칭

**파일**: `backend/src/services/routeService.ts:21-22`

**현상**:
ODsay `startName`과 BusanBIMS `bstopnm`이 완전히 일치하지 않을 수 있다.
(예: ODsay는 "서면역", BusanBIMS는 "서면역.서면지하상가")
`busStopList?bstopnm=서면역`으로 조회 시 BusanBIMS가 부분 매칭을 지원하는지 미검증.

**영향**: 일부 정류소에서 `bstopid = null` → fallback 5분 처리

## 4. 코드 상태 점검

### 4-1. 알고리즘 구현 상태

| 지표 | 구현 방식 | 상태 |
|------|-----------|------|
| $S_{time}$ | ODsay `totalTime` → 20~120분 정규화 | ✅ 정상 |
| $S_{transfer}$ | ODsay `busTransitCount + subwayTransitCount` 계단형 패널티 | ✅ 정상 |
| $S_{walk}$ | ODsay `totalWalk` → 0~1200m 정규화 | ✅ 정상 |
| $S_{wait}$ | BusanBIMS `min1` 실시간 → 0~20분 정규화 | ✅ 정상 (버스 경로 한정) |
| $S_{access}$ | DB `accessScore` → 직접 사용 | ⚠️ UUID 불일치로 미반영 |

### 4-2. 최종 점수 수식 검증

$$\text{Final Score} = 100 \times (0.45 \cdot S_{time} + 0.20 \cdot S_{transfer} + 0.15 \cdot S_{walk} + 0.10 \cdot S_{wait} + 0.10 \cdot S_{access})$$

실제 계산값 역산 검증 (감천문화마을, isFallback=false):
```
totalTimeMin=30 → s_time = 1-(30-20)/100 = 0.90 ✅
transferCount=0  → s_transfer = 1.00          ✅
walkDistanceM=467 → s_walk = 1-467/1200 = 0.61 ✅
waitTimeMin=1    → s_wait = 1-1/20 = 0.95      ✅
accessScore=0.5  → s_access = 0.50 (UUID 불일치)
finalScore = 100*(0.45*0.90+0.20*1.0+0.15*0.61+0.10*0.95+0.10*0.50)
           = 100*(0.405+0.200+0.0915+0.095+0.050) = 84.15 ≈ 83.7 ✅
```

수식 자체는 정확히 동작함.
